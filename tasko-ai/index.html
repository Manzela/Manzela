Operational Context

The Dispatch Bottleneck

Hasherut operated a Human-as-a-Service task platform via WhatsApp. Every client request required a human dispatcher to read, classify, price, and assign — and the institutional knowledge lived entirely in chat history.

forum

WhatsApp-Native Workflow

Clients messaged the company directly on WhatsApp; taskers fulfilled. Every assignment passed through a human dispatcher reading messages in real time — no ticketing system, no structured intake form.

groups

Manual Classification at Scale

200+ active client threads across 180+ distinct task types — driving, deliveries, errands, repairs, event staffing, pet care, personal shopping. Classification, pricing, and assignment all sat in one person’s head.

history_edu

No Structured Memory

Nine years of operational knowledge — pricing logic, escalation paths, edge-case handling — was trapped in conversational chat exports. No database. No knowledge base. Just text.

target

Engineering objective. Convert nine years of unstructured Hebrew/English WhatsApp conversation into a structured training corpus, fine-tune a language model on it, and put that model behind the same WhatsApp number — replacing the dispatcher loop with a model-in-the-loop conversational layer.

תהליכים ונהלים

Hebrew operational documentation

Internal standard operating procedures, originally authored in Hebrew, converted into a retrievable knowledge layer. Preserved the source language so retrieval matched the language of the live conversation rather than introducing a translation step at inference.

policy

Why retrieval, not just fine-tuning. The chatbot was constrained by retrieval against company-authored rules — preventing free-form generation that would violate operational policy. The fine-tuned weights handled style and shape of replies; the RAG corpus handled truth.

System Design

Serverless Conversational Layer

WhatsApp messages enter via Twilio, route to AWS Lambda, query the fine-tuned OpenAI model with retrieval against the operational corpus, and return synchronously inside the same WhatsApp thread.

WhatsApp→ Twilio Webhook→ AWS Lambda→ OpenAI (fine-tuned)→ Lambda→ Twilio→ WhatsApp
phone_iphone

WhatsApp Gateway

Twilio WhatsApp Business API. Inbound messages arrive at a webhook endpoint; outbound replies post back through the same number, preserving the existing customer surface.

bolt

Compute

AWS Lambda (Node.js). Stateless functions handle Twilio webhook payloads, orchestrate retrieval, and call the model API. Pay-per-invocation matched the conversational workload.

smart_toy

AI Backend

Fine-tuned OpenAI model for in-domain replies, with ChatGPT used for out-of-distribution fallbacks during evaluation cycles.

menu_book

RAG Layer

Retrieval against the workflows / processes corpus (תהליכים ונהלים) at request time — constraining replies to the company’s authored operational rules.

web

CRM Surface

Wix-based company website with embedded chat as a secondary surface alongside the WhatsApp channel.

groups_3

Operations

Connecteam for tasker management and shift coordination — the human side of the workflow that the chatbot was designed to interleave with, not replace.

architecture

Why serverless. Lambda + Twilio kept the production surface a thin proxy — no servers to maintain, no idle cost between conversations, and a clean re-deploy path each time a new fine-tuned model checkpoint replaced its predecessor.

Approaches Evaluated

Three Architectures, One Production Path

Three chatbot architectures were evaluated before committing to the Lambda + OpenAI serverless stack. Each ruled-out path produced a concrete constraint that shaped the next.

APPROACH 01

Dante AI

No-code chatbot platform with a native Twilio WhatsApp connection. Evaluated for time-to-deploy and the strength of its drag-and-drop flow editor.

Outcome: Limited customization for the Hebrew-domain retrieval pattern and the SQuAD-style training data already accumulated.

APPROACH 02

AutoTrain + ChatGPT

Hugging Face AutoTrain for model training, ChatGPT for inference. Iterated through dataset versions testing_3 – testing_6.

Outcome: Data quality, not the model, was the bottleneck. This is what drove the SQuAD-format Q&A pipeline (Stage 04).

APPROACH 03

AWS Lambda + OpenAI

Custom serverless function bridging Twilio webhooks to a fine-tuned OpenAI model with retrieval against the operational corpus — the production stack.

Outcome: Selected as the production path; clean re-deploy cycle per new fine-tuned checkpoint.
